# Obsidian 用户操作指南

Agent Task Loop（ATL）可以独立完成任务收集、去重、确认和受控调研。推荐与 TaskNotes 配合使用：TaskNotes 提供任务卡片和看板，ATL 负责任务闭环。ATL 不是 TaskNotes 的 fork，也不会修改 TaskNotes 插件文件。

## 安装插件

1. 从项目 [Releases](https://github.com/PANGKAIFENG/agent-task-loop/releases) 下载 `agent-task-loop-v0.3.1.zip`。
2. 解压后确认有 `main.js`、`manifest.json`、`styles.css`、`atl-runner.mjs` 四个文件。
3. 在 Finder 中打开 Vault，按 `Command + Shift + .` 显示隐藏文件。
4. 进入 `.obsidian/plugins/agent-task-loop/`；没有该文件夹时直接新建。
5. 把四个文件放入该文件夹，重启 Obsidian。
6. 打开“设置 → 第三方插件”，启用 `Agent Task Loop`。

安装 ATL 本身不需要终端，也不需要下载源代码。后台调研依赖本机已有 Node.js 24+ 和已登录的 Claude Code；ATL 会检测并连接它们。

## 第一次配置

打开“设置 → Agent Task Loop”，按页面顺序完成三部分。

### 1. Vault 权限

开启“允许 ATL 管理此 Vault”。授权默认关闭；开启后 ATL 可以确认和移动任务、写入执行结果、管理自己的后台配置。

ATL 只管理当前 Vault 的 `10_Tasks` 任务数据和自己的后台配置，不会修改其他 Obsidian 笔记。

### 2. 后台执行

页面会自动检查四项：

- `ATL Runner`：是否已随插件正确安装；
- `Node.js 24+`：是否找到可用 Node.js；
- `Claude Code`：是否找到且已登录；
- `后台任务`：ATL 的定时任务是否已安装。

在“模型服务”中选择一种运行方式：

| 模式 | 适合场景 | ATL 会保存什么 |
| --- | --- | --- |
| 沿用 Claude Code 当前配置 | 已经在 Claude Code 或 CC-Switch 中管理模型服务，推荐 | 不覆盖 Model 或 Base URL |
| 自定义服务 | 只想让 ATL 使用另一个 Claude Code 兼容服务 | 仅保存 Model 与 Base URL |

选择“自定义服务”后必须填写：

- `Model`：服务支持的模型标识，例如 `glm-4-flash`；
- `Base URL`：完整的 `http` 或 `https` 地址，例如 `https://api.example.com/anthropic`。

ATL 不提供 API Key 输入框，也不会把密钥写入 Vault、插件设置或后台任务。密钥和登录仍由 Claude Code 或系统环境管理。自定义服务会收到任务目标和已授权资料，只使用你信任的 endpoint。包含账号、密码、查询参数、片段或非 `http/https` 协议的地址会被拒绝。

修改模型服务后，已安装的后台任务不会被静默改写。点击“更新后台配置”后新设置才会生效。

用“选择文件夹”添加 Agent 可以读取的本地资料目录。ATL 不接受在设置中输入任意命令，只使用系统文件夹选择器保存明确授权的路径。

常见状态：

| 状态 | 操作 |
| --- | --- |
| 未配置 | 等待自动检测，或补齐页面显示缺失的外部环境 |
| 待安装 | 点击“启用后台执行” |
| 已就绪 | 配置完成，可等待整点运行或点击“立即试跑” |
| 正在执行 | 等待本次检查完成 |
| 配置异常 | 按提示处理 Claude 登录、无效路径或后台配置冲突，再点“重新检测” |

启用后，ATL 会在每天 `08:00` 至 `22:00` 的每个整点检查一次队列。停用时点击后台操作右侧的电源图标；ATL 只移除自己管理的后台任务。

### 3. 任务看板

- “ATL 紧凑卡片”：让 TaskNotes 卡片突出标题、确认状态、来源日期和优先级。
- “应用推荐布局”：把任务总看板切换为适合 ATL 的紧凑布局，并隐藏冗余字段。
- “恢复原布局”：恢复首次应用前保存的原始 Base 文件。

ATL 首次应用推荐布局时会创建 `.atl-backup`，已有备份不会被覆盖。

## 随手新建任务

1. 点击 Obsidian 左侧 Ribbon 的“ATL：新建任务”。
2. 填写标题，可选填写补充说明和优先级。
3. 点击“加入 Inbox”。

这个入口适合当场记录，不要求你立即想清项目、目标和验收标准。这些信息在后续确认任务时补齐。

## 从笔记同步助手立即获取待办

1. 先将想法或待办发给“笔记同步助手”，等它出现在当前 Vault 的 `笔记同步助手/` 目录中。
2. 点击左侧 Ribbon 的“ATL：从同步助手获取待办”。
3. ATL 调用已配置的 Claude Code 或 CC-Switch 模型服务提取候选。
4. 在居中弹窗中检查候选；默认全选，可以取消任意一项，也可以全部忽略。
5. 点击“将所选任务加入 Inbox”。

ATL 只读同步助手原文，不会移动、删除或改写来源文件。首次使用时扫描昨天和今天；之后从上次成功扫描所在的本地日期补扫到今天。

只有点击提交且处理成功后，ATL 才推进扫描进度。直接取消弹窗、模型失败或任务写入失败时，不会跳过这批内容。明确取消勾选并提交的候选会记为已审阅，下次不再重复提示。

### 与每日复盘如何去重

已配置的每日复盘和这个实时扫描入口都通过 ATL 的同一收集服务创建任务：

1. 先检查稳定的来源标识；
2. 再比较来源笔记、任务标题和原文证据；
3. 确认是同一行动时返回已有任务，不创建第二条；
4. 只有标题相似而证据不足时，保留两条并标记“疑似重复”。

因此，无论先跑每日复盘，还是先点实时扫描，同一来源的同一行动最终都只保留一个任务。

### 常见处理

- 未检测到“笔记同步助手”：先确认同步助手已在当前 Vault 中生成目录。
- 未发现新记录：当前扫描窗口没有新的未审阅内容，无需处理。
- 本次没有待办候选：模型已检查新内容，但没有找到值得进入 Inbox 的行动。
- 模型或扫描失败：按提示到 ATL 设置检查 Claude Code、Model 和 Base URL；修复后直接再点一次，不需要终端。

## 日常确认任务

1. 打开 TaskNotes 的任务总看板。
2. 在“收件箱”中找到需要处理的任务。
3. 打开任务卡片的文件菜单，选择“确认并移到待执行”。
4. 在居中弹窗中选择项目，或创建新项目。
5. 填写任务目标和至少一条验收标准。
6. 设置优先级。
7. 决定是否开启“允许 Agent 自动执行”。
8. 点击“确认并移到待执行”。

确认成功后，任务进入“待执行”。关闭 Agent 自动执行时，任务仍可进入待执行，但后台调度器不会领取它。

如果任务文件已经打开，也可以从 Obsidian 命令面板运行“Agent Task Loop: 确认当前任务并移到待执行”。

## 项目怎么选

- **选择已有项目**：适合同一产品、研究主题或持续目标下的任务。
- **创建新项目**：只有当任务属于一个新的长期目标时再创建；填写项目名称和简短说明即可。
- **不要每项任务都建项目**：项目用于共享上下文和聚合结果，不是任务分类标签。

示例：

| 任务 | 建议项目 |
| --- | --- |
| 对比三款 Obsidian 任务插件 | 个人任务系统 |
| 调研 FDE 岗位要求 | 职业发展研究 |
| 补充产品健康度用户场景 | 产品健康度看板 |

## 目标和验收标准怎么写

目标说明“这次最终要回答什么”，验收标准说明“看到什么结果才算完成”。

```text
任务：调研 FDE 角色与 AI 产品经理的适配度

目标：明确 FDE 的核心职责、能力要求，以及它与我当前 AI 产品经理经验的匹配点和差距。

验收标准：
1. 至少引用 3 个官方招聘页面或公司岗位说明。
2. 输出职责、能力、匹配点和差距四部分。
3. 明确区分公开事实、分析判断和待验证信息。
4. 给出 3 个可执行的下一步行动。
```

好的验收标准应当可检查，例如“引用至少一个官方页面”；避免使用“深入研究”“全面分析”“尽量详细”这类无法判断是否完成的表达。

## Agent 会处理哪些任务

后台只会领取同时满足以下条件的任务：

- 状态是 Ready；
- 类型是 `research`；
- 权限是 `read_only_research`；
- 已关联项目；
- 已填写目标和至少一条验收标准；
- 你主动允许 Agent 自动执行。

“立即试跑”也遵守同一规则。没有合格任务时会安全结束，不会自动确认 Inbox 任务，也不会为了测试而移动个人任务。

## 如何验收结果

任务执行后会进入 Review，并关联一个 Artifact。验收时重点检查：

1. 结论是否回答了任务目标；
2. 每条验收标准是否有对应说明；
3. 关键事实是否有可访问的证据；
4. 事实、推断和未知项是否区分清楚；
5. 是否需要带反馈退回、阻塞或取消。

Agent 不能批准自己的结果。只有你确认后，任务才会进入 Done。

## 常见提示

- **“请先允许 ATL 管理此 Vault”**：到插件设置开启 Vault 权限。
- **“任务已经不在收件箱”**：TaskNotes 可能尚未刷新，重新打开看板。
- **“同名项目已经存在”**：重新打开弹窗，从已有项目中选择。
- **“任务刚刚被其他操作修改”**：刷新任务后再确认，ATL 不会覆盖并行修改。
- **Node.js 未找到或版本过低**：安装 Node.js 24+ 后回到设置重新检测。
- **Claude Code 未登录**：先在 Claude Code 中完成登录，再重新检测。
- **Model 或 Base URL 无效**：补齐自定义服务字段；Base URL 必须是可信的完整 `http/https` 地址。
- **后台配置冲突**：ATL 检测到同名但不受它管理的 macOS 服务，因此不会覆盖；需要开发者检查后处理。

## 当前边界

`v0.3.1` 已把 ATL Runner 和定时任务的连接、启用、试跑、更新与停用放进 Obsidian，并增加随手新建和同步助手实时待办提取。Node.js 和 Claude Code 仍是外部前提，ATL 不负责安装或登录第三方软件。任务智能扩写、Agent/Skill/小队编排和完整的 Obsidian 内验收工作台不在本期范围。
