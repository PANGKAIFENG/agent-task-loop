# ATL 人工优先任务管理设计规格

## 1. 背景

ATL 当前把 Obsidian 任务默认建模为可由 Agent 执行的调研任务。用户把 Inbox 任务移入 Ready 前，必须选择项目、填写目标与验收标准，并确认只读调研权限。该约束保证了无人值守执行安全，却让最常见的人工任务管理变得过重。

用户当前首先需要一个可长期使用的个人任务系统：从笔记同步助手和人工输入收集待办，在 Obsidian 看板中自由推进状态，通过 TaskNotes 日历安排时间；只有在确实需要时，才让 AI 帮助整理任务或把任务交给 Codex。

## 2. 问题与目标

### 当前问题

- 人工任务进入“待办”被 Agent 执行字段阻塞。
- 看板把七个 ATL 内部生命周期状态直接暴露给用户，主流程过重。
- 看板列由硬编码公式生成，不能自然跟随 TaskNotes 自定义状态。
- 用户需要自己填写目标、验收标准和权限，AI 没有承担任务澄清工作。
- TaskNotes 的自由状态与 ATL Runner 的执行状态共用一个字段，边界不清晰。

### 设计目标

1. 默认主视图只有“收件箱、待办、进行中、已完成”四列。
2. 状态名称、颜色、顺序和新增状态继续由 TaskNotes 管理，ATL 不锁死列数和中文名称。
3. 普通任务只需标题即可进入待办并由用户人工推进。
4. AI 整理是可选动作：用户用一句自然语言补充意图，AI 生成任务目标与可判断的完成条件。
5. AI 不可用或未配置时，人工任务管理仍完整可用。
6. 保留现有受控调研 Runner，但只有显式授权且信息完整的任务才可被自动领取。
7. 用户可把任务整理为 Codex 可理解的上下文，继续在 Codex 中人工完成。

## 3. 用户与角色

- **个人用户**：收集任务、调整状态、安排时间、完成任务。
- **AI 整理助手**：根据任务标题、已有说明和用户补充，生成结构化建议，不自动执行任务。
- **ATL Runner**：只领取显式开启自动执行且满足安全准入条件的调研任务。
- **TaskNotes**：提供状态配置、看板拖动、任务卡片和日历。

## 4. 约束

- Markdown 是任务数据的唯一事实源。
- TaskNotes 是可选的可视化层；ATL 不 fork TaskNotes、不操作其 DOM，也不调用未公开的私有 API。
- ATL 不自动改写用户的 TaskNotes 全局状态配置。
- 插件读写能力限于当前 Vault 的 `10_Tasks` 与自身配置；同步助手来源保持只读。
- MVP 以 macOS 桌面版 Obsidian 为主，不要求用户打开终端。
- 现有任务、CLI、Runner、审计和归档数据必须向后兼容。

## 5. 方案比较

### 方案 A：自由人工状态，Agent 准入独立判断（采用）

- TaskNotes 管理任意状态值；ATL Core 能读取并保留自定义状态。
- 四个推荐状态仅作为默认视图，不是固定枚举。
- Runner 仍只识别 ATL 的受控执行状态，并额外检查自动执行授权与任务完整度。
- 优点：人工管理自由，保留现有安全边界，改动可渐进落地。
- 代价：自定义状态不自动获得 Runner 语义；用户需要显式执行“交给 AI”。

### 方案 B：只允许四个固定状态

- 实现最小，但违背用户希望自由命名和增加状态的需求。
- 不采用。

### 方案 C：立即建立完整双状态机

- 新增独立 `workflow_status` 与 `agent_state`，迁移全部历史任务与接口。
- 长期边界最清楚，但 MVP 迁移面和回归风险过大。
- 本期只建立兼容边界，不进行完整数据模型迁移；后续自动化增强时再演进。

## 6. 推荐方案与设计

### 6.1 人工看板

- 推荐 Base 视图直接按 Markdown 的 `status` 分组，不再通过七状态中文公式分组。
- 默认固定显示 `inbox`、`ready`、`in_progress`、`done` 四个落点。
- 展示名称由 TaskNotes `customStatuses` 决定，推荐中文为“收件箱、待办、进行中、已完成”。
- 空的扩展状态默认隐藏；用户在 TaskNotes 中新增状态后，状态一旦被使用即可出现在看板，也可由用户在视图设置中固定。
- 拖动由 TaskNotes 原生能力负责。ATL 不拦截拖动，也不依赖显示名称判断状态。

### 6.2 人工进入待办

- Inbox 卡片菜单文案调整为“移到待办”。
- 居中弹窗保留任务标题，项目、目标和完成条件全部可选。
- 用户可以不填写任何高级字段，直接把任务状态改为 `ready`。
- `auto_executable` 默认并保持 `false`，因此普通待办不会被后台 Runner 领取。
- 项目为空时任务进入 `10_Tasks/Active/unassigned`；状态仍由 Markdown 决定。

### 6.3 自定义状态兼容

- `Task.status` 从七值封闭枚举调整为经过长度和控制字符校验的非空字符串。
- ATL 内部仍保留七个已知执行状态常量，用于已有 CLI、Runner 和审计流程。
- 未知状态可以读取、索引、查询和原样保存，但不能被 Runner 自动领取。
- 未知状态默认存放在 Active 生命周期目录；ATL 不根据自定义中文名称推断完成或归档语义。
- TaskNotes 写入的 `scheduled`、`due`、完成日期等未知 frontmatter 字段在 ATL 保存任务时必须保留。

### 6.4 AI 帮我整理

- 弹窗提供一个可选的“告诉 AI 你想完成什么”输入区和“AI 帮我整理”按钮。
- AI 输入只包含任务标题、任务正文、用户补充和已有项目名称；不读取其他 Vault 文件。
- AI 返回：建议的任务目标、1 至 5 条完成条件；项目和优先级仍由用户决定，避免模型擅自分类。
- 生成结果回填到可编辑字段，只有用户点击“移到待办”后才写入 Markdown。
- AI 失败、超时或模型未配置时保留用户输入并显示可恢复错误；“移到待办”始终可用。
- 复用现有 Claude Code 检测、Model 和 Base URL 配置，不新增第二套密钥存储。

### 6.5 交给 Codex

- 所有 ATL 任务文件的 Obsidian 文件菜单提供“复制给 Codex”。
- 复制内容包含绝对任务文件路径、标题、正文、项目、目标、完成条件和来源摘要；空字段明确标记为“未设置”。
- 该动作只复制上下文，不启动外部进程、不发送网络请求，也不改变任务状态。

### 6.6 Agent 自动执行边界

- `ready` 只表示人工看板中的“待办”，不再等价于“可自动执行”。
- Runner 继续要求：`status=ready`、`review_state=confirmed`、`auto_executable=true`，以及项目、调研类型、目标、完成条件和只读权限全部有效。
- 本期的轻量任务弹窗不默认展示自动执行开关。
- 已有 CLI 和高级流程仍可补齐完整字段并显式开启自动执行。

## 7. 核心流程

### 收集与人工管理

1. 用户手动新建任务，或从同步助手实时/每日收集任务。
2. 去重后的任务进入 `inbox`。
3. 用户在看板拖动，或通过文件菜单打开居中弹窗。
4. 用户可直接移到待办，也可补充项目、目标和完成条件。
5. TaskNotes 根据 `scheduled` 与 `due` 在日历展示任务。
6. 用户继续拖动任务到进行中或已完成。

### AI 整理

1. 用户打开 Inbox 任务弹窗。
2. 输入一句补充，例如“我想判断它值不值得接入 StyleWork”。
3. 点击“AI 帮我整理”。
4. AI 返回目标与完成条件，弹窗回填。
5. 用户编辑或直接确认，任务进入待办。

### Codex 执行

1. 用户在任意任务卡片的文件菜单选择“复制给 Codex”。
2. 插件复制结构化任务上下文。
3. 用户在 Codex 中粘贴并主动开始工作。
4. 任务状态仍由用户在 Obsidian 中管理。

## 8. 数据模型

### 本期保留字段

- `status: string`
- `review_state`
- `project_id`
- `objective`
- `acceptance_criteria`
- `auto_executable`
- `permission_profile`
- 现有来源、审计、Claim 和 Artifact 字段

### TaskNotes 字段

- `scheduled`
- `due`
- `timeEstimate` 或用户当前 TaskNotes 映射字段
- 完成日期字段

ATL 不消费这些日历字段，但保存任务时必须保留它们。

## 9. UI 与交互

- 使用现有 Obsidian 居中 Modal，不创建独立网页表单。
- 首屏只展示任务标题、项目、优先级和可选的 AI 补充输入。
- 目标与完成条件放在“任务说明”区域；AI 回填后自动展开。
- 主按钮为“移到待办”，不再使用“确认并移到待执行”。
- AI 按钮有明确的忙碌、成功和失败反馈，忙碌期间不清空表单。
- 看板卡片优先显示标题、项目、计划时间、截止时间和优先级；Agent 字段不再占据主卡片信息层级。

## 10. 异常与人工介入

- TaskNotes 未安装：ATL 收集、弹窗和复制给 Codex仍可用；看板与日历不可用。
- 自定义状态格式无效：ATL 拒绝解析并报告字段错误，不回写文件。
- AI 环境不可用：只禁用本次 AI 整理，不阻塞人工推进。
- AI 返回非法结构：显示“未生成可用建议”，保留已有内容。
- TaskNotes 与 ATL 同时修改同一任务：沿用快照冲突检测，提示刷新后重试。
- 自定义完成状态：TaskNotes 可以显示为完成；ATL 不自动把它解释为 `done` 或触发归档。

## 11. 不做范围

- 不开发新的看板、日历或 TaskNotes fork。
- 不自动修改 TaskNotes 私有配置文件。
- 不实现多轮聊天、自动拆解子任务或自动创建项目。
- 不实现无人值守领取任意人工任务。
- 不实现 Agent、小队、Skill 编排和跨设备云执行。
- 不实现移动端本地 Runner。

## 12. 验收标准

1. 推荐看板按原始 `status` 分组，并默认固定显示四个主状态。
2. 用户在 TaskNotes 改状态显示名称或添加新状态后，ATL 不会因未知状态而拒绝读取任务。
3. Inbox 任务不填写项目、目标、完成条件或权限也能移到待办。
4. 该任务默认不会被 Runner 自动领取。
5. 现有完整且显式授权的调研任务仍可被 Runner 执行。
6. “AI 帮我整理”能根据标题、正文和一句补充生成目标及完成条件，并允许用户编辑后保存。
7. AI 不可用时仍能直接移到待办。
8. 任意 ATL 任务可复制为 Codex 上下文，复制动作不修改任务。
9. TaskNotes 日历相关未知 frontmatter 在 ATL 读取和保存后保持不变。
10. 插件构建产物可安装到当前 Vault，Obsidian 中不需要终端操作。
11. 原有实时收集、每日去重、CLI、Runner、审计和 Artifact 流程回归通过。

## 13. 主要风险

- TaskNotes 对 `pinnedColumns` 与自定义状态显示的细节可能随版本变化；只使用 Base 文件公开格式并保留备份。
- 放宽 `status` 类型后，所有依赖穷举状态的代码必须显式处理未知状态，避免任务在自带 Web 看板中消失。
- AI 整理复用 Claude Code，会受本机登录、模型兼容性和 Base URL 配置影响，因此必须保持人工降级路径。
- 自定义完成状态不会自动触发 ATL 归档；这是自由状态与安全执行解耦的明确取舍。

## 14. 待确认与开放问题

无阻塞开放问题。多轮 AI 澄清、自动权限推断和完整双状态机留待二期重新设计。
